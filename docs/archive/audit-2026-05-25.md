## 0. Шкала эффекта

- **Критичный** — убирает риск молчаливой потери сигналов, зависших worker'ов, частично примененного persistent state или невозможности нормально остановить сервис.
- **Высокий** — оператор или reviewer сразу видит разницу: мониторинг, доказуемая архитектура, понятная диагностика.
- **Средний** — улучшает сопровождаемость и скорость будущих изменений.
- **Опциональный** — делать после Waves 1-4, если есть отдельная мотивация.

## 1. Что уже хорошо

- Граница adapter/core сильная: PyMax knowledge живет под `src/adapters/max/backends/pymax/`, aiogram — под TG adapter, `BridgeCore` работает через transport-neutral contracts.
- PyMax 2 migration выполнена с хирургическими shim'ами: legacy session import, reconnect/telemetry flags, msgpack key wrapping, TCP seq wrapping, UNSUPPORTED strip, MAX-only egress socket injection.
- Есть ADRs, bilingual README, runbooks, Ansible deploy/backup/recover, CI с ruff/mypy/pytest/ansible-lint.
- Privacy posture правильная: тексты сообщений и медиа не логируются и не хранятся в DB, кроме временных durable text retry queues до delivery/TTL.

Главное правило: не ломать boundary. При новой MAX-фиче не импортировать pymax-типы в core; расширять contracts/DTO.

## 2. Corrections From Follow-Up Review

- PyMax surface использует `pymax.connection` и `pymax.connection.readers`, не `pymax.conn`. Surface-pin тест должен строиться по фактическим imports текущего backend.
- Logging уже semi-structured через `EventFormatter`, `log_event`, ordered event fields и `LOG_FORMAT=text|json|mixed`. Первый шаг — расширить текущий слой, а не обязательно внедрять `structlog`.
- SQLite migrations уже версионированы как `Migration(1..3)`. Нельзя описывать runner как one-shot; полезный follow-up — migration immutability policy/test.
- Backup уже включает env-файлы, `config.local.yaml` и `data/`. Gap — documented restore drill, а не basic backup inclusion.
- В repo нет Ansible `monitoring`/`node_exporter` роли. Metrics textfile path должен быть configurable и по умолчанию безопасно писать в `data/maxtg_bridge.prom`.
- Restore-drill на новом VPS сейчас не выполняем, потому что spare VPS нет; документируем checklist с `not yet executed`.

## 3. Wave 1 — production-readiness

### 3.1 Graceful shutdown

Эффект: `docker stop`/SIGTERM не должен отменять in-flight tasks без drain и закрытия ресурсов.

Изменения:

- `src/main.py`: добавить `SIGTERM`/`SIGINT` handlers через `loop.add_signal_handler`, `stop_event`, structured log `app.shutdown.requested`.
- `src/runtime/supervisor.py`: `run(stop_event=...)`, worker task cancellation, heartbeat cancellation, final heartbeat write.
- `src/startup/composition.py`: гарантированно закрывать TG adapter, Repository и MAX adapter; не слать crash alert на intentional shutdown.
- Tests: supervisor graceful stop, composition close-order smoke.

### 3.2 Cross-repo transactions

Эффект: clustered DB writes после внешнего send не должны частично коммититься.

Правило: не держать SQLite transaction вокруг TG/MAX network await. Inbound idempotency key остается отдельным pre-send commit.

Изменения:

- `Repository.transaction()` с `BEGIN IMMEDIATE` / `COMMIT` / `ROLLBACK`.
- `BaseRepo._commit()` пропускает commit внутри активной transaction.
- Использовать transaction для post-send groups: inbound mapping + delivery, queued retry + delivery, outbound mapping + delivery.
- Tests: rollback при ошибке между DB writes; nested transaction запрещена понятной ошибкой.

### 3.3 Detached tasks

Эффект: exceptions из fire-and-forget tasks не теряются.

Изменения:

- Добавить `src/runtime/tasks.py` с `create_logged_task(...)` / `attach_task_logger(...)`.
- Применить к outbox task, recovery event scan tasks, voice recovery workers и heartbeat/owner tasks, которые не await'ятся владельцем.
- Tests: task exception логируется с именем task и traceback.

### 3.4 External await timeouts

Эффект: зависший TG/MAX/socket await не держит worker вечно.

Изменения:

- Добавить helper `with_timeout(awaitable, timeout, label)` или узкие wrappers в bridge layer.
- Defaults: 30s для send/topic operations, 120s для media transfer path.
- На timeout логировать metadata-only event и переводить в существующий retry/failure path.
- Tests: hung TG send queues/marks failure according to existing retry policy.

### 3.5 Restore-drill runbook

Эффект: следующий оператор знает, как проверить disaster recovery, даже без spare VPS сейчас.

Изменения:

- Добавить `docs/runbooks/disaster-recovery-drill.md` с bootstrap/recover checklist, expected evidence и статусом `not yet executed`.
- Не писать фиктивное время восстановления.

## 4. Wave 2 — observability

### 4.1 Event vocabulary поверх текущего logging

Эффект: `LOG_FORMAT=json` + stable events дают grep/jq диагностику без новой logging framework migration.

Итог внедрения:

- Отдельный `src/logging_events.py` пока не добавлен: текущий `log_event`/`EventFormatter`
  уже дают ordered `event_fields`, а механическая замена всех literals на constants
  не даёт достаточного эффекта для этого шага.
- В новых reliability/metrics путях используются stable event names и metadata-only
  fields; без msg text/raw payload.
- Docs: `docs/runbooks/operations.md` явно рекомендует `LOG_FORMAT=json` для machine-readable diagnostics.

### 4.2 Prometheus textfile metrics

Эффект: health/retry state можно мониторить снаружи без HTTP endpoint.

Изменения:

- Новый renderer/writer в runtime health/metrics layer.
- Default output: `data/maxtg_bridge.prom`.
- Optional override через env/config для node_exporter textfile directory.
- Метрики: MAX connected, pending queues, delivery totals by direction/status, worker restarts, last healthy timestamp, alert outbox size.
- Atomic write: tmp + rename.
- Tests: renderer format, atomic replacement, disabled/missing path behavior.

### 4.3 Alert dedup review

Эффект: long outage не должен спамить owner DM.

Изменения:

- Проверить текущий `alert_outbox`/health issue signature behavior.
- Если dedup уже есть через `reason_changed`/reminder interval, задокументировать.
- Если нет — добавить sliding-window dedup по issue signature/category.

### 4.4 Supervisor backoff

Эффект: network flap не дает сотни restart/alert cycles в час.

Итог внедрения:

- Используется текущий `worker_restart_backoff_seconds` как base.
- Delay: exponential with 50-150% jitter, cap 300s.
- Tests: sequence растет и cap соблюдается; alert text показывает текущий delay.

## 5. Wave 3 — architecture proof / reviewer signal

### 5.1 FakeMaxBackend integration proof

Эффект: ADR-007 "backend replaceable" становится CI-фактом.

Изменения:

- `tests/fakes/fake_max_backend.py`: in-memory `MaxBackend`/`MaxClientPort`, no PyMax imports.
- `tests/integration/test_bridge_end_to_end.py`: fake MAX message -> TG topic/message -> TG reply -> fake MAX send capture.
- `examples/swap_max_backend.py`: runnable demo without credentials.
- Fake payloads metadata-only: no real message text, phones, invite links, tokens or raw MAX payloads.

### 5.2 PyMax surface-pin

Эффект: upstream PyMax drift ловится в CI до deploy.

Изменения:

- `tests/test_pymax_surface_pin.py` imports фактических backend symbols:
  `pymax`, `pymax.client`, `pymax.connection`, `pymax.connection.readers`,
  `pymax.protocol`, `pymax.protocol.tcp`, framing/payload, transport, session,
  auth/session/message payload modules.
- Mark `pytest.mark.architecture`.
- Failure message: upstream surface changed, adjust backend before deploy.

### 5.3 README / ADR packaging

Эффект: reviewer за 5 минут видит реальную сложность проекта.

Изменения:

- README/README-ru: текущая PyMax v2 shim story в начало; v1 OOM/reconnect — historical note.
- `docs/decisions/ADR-010-pymax-v2-migration.md`: decision, rejected alternatives, consequences, regression guards, link to migration journal.
- Public docs держать bilingual без смыслового drift.
